# 版本规划与交付节奏

## 1. 规划原则

1. 先稳定主链路，再扩展专业能力，最后开放生态能力。
2. 每个阶段都必须包含“功能 + 性能 + 验收 + 文档回写”。

## 2. 里程碑分期

## M1（P0）主链路稳定化

目标：

1. 核心交互主链路稳定可用。
2. 高频交互性能可控。

范围：

1. 模块 01 全量
2. 模块 02 的变换与吸附主链路
3. 模块 05 的命令桥与诊断基础

交付物：

1. 统一状态机与命令映射文档
2. 高频场景性能基线
3. 核心链路回归清单

退出标准：

1. 核心任务可稳定完成。
2. 关键性能指标达到基线目标。

## M2（P1）专业编辑深度

目标：

1. 强化路径、文字、样式、结构能力。

范围：

1. 模块 02 路径增强
2. 模块 03 文字/样式增强
3. 模块 04 结构交互增强

交付物：

1. 高级编辑能力验收集
2. 属性面板一致性规范

退出标准：

1. 复杂编辑任务通过率提升。
2. 设计师内测满意度达到目标。

当前状态（2026-05-21）：

1. M2 已启动：开工任务清单与目标期望已建立。
2. 执行入口：
   - `docs/product-requirements/m2-execution-backlog.md`
   - `docs/product-requirements/m2-target-expectations.md`

## M3（P2）平台化能力

目标：

1. 建立组件、协作、扩展基础能力。

范围：

1. 模块 06 全量（分批）
2. 模块 05 可观测性增强

交付物：

1. 组件系统 MVP
2. 协作 MVP
3. 插件 API Alpha

退出标准：

1. 多人协作主链路可演示。
2. 组件复用链路可用于真实项目。

## 3. 优先级建议

P0（必须先做）：

1. 选择/命中/变换稳定性
2. 吸附与对齐可预测
3. 预览提交一致性
4. 性能与诊断基线

P1（紧随其后）：

1. 路径编辑增强
2. 文字高级排版
3. 图层结构与布尔体验增强

P2（战略扩展）：

1. 组件资产库
2. 协作能力
3. 插件生态

## 4. 依赖关系

1. 组件系统依赖：稳定对象 ID 与变更追踪。
2. 协作系统依赖：命令可回放与冲突策略。
3. 插件系统依赖：命令接口稳定与权限模型。

## 5. 版本治理机制

1. 每个迭代必须更新：

- 已完成需求
- 未完成原因
- 风险与下一步

2. 每个里程碑结束执行一次：

- 功能验收
- 性能回归
- 文档一致性检查

## 6. 当前建议执行顺序

1. 立即进入 M1：修正高频交互一致性与性能稳定性。
2. 并行预研 M2 文本与路径增强方案。
3. 在 M1 稳定后启动 M3 的组件系统 RFC。

## 7. 立即执行（M1 开工版）

阶段 S0（本周，治理与基线）：

1. 固化架构决策：Product 与 Runtime 不合并，按契约协作。
2. 建立 M1 任务看板与验收口径。
3. 补齐高频交互基线数据采样（选择、拖拽、变换、路径编辑）。

阶段 S1（第 1 周，核心链路稳定）：

1. 统一 pointer 生命周期与状态机切换边界。
2. 统一命中优先级与选择策略入口。
3. 修正预览态与提交态一致性问题。

阶段 S2（第 2 周，性能与诊断）：

1. 吸附预算与命中预算策略收敛。
2. 建立关键路径性能报表（P50/P95）。
3. 增加异常中断恢复与回滚回归用例。

执行清单：

1. 详见 m1-execution-backlog.md。
