# Render Parity Runtime Auto Reports

This folder stores persisted artifacts generated by the runtime auto parity command.

## Command

pnpm run report:render-parity-runtime-auto -- --input <runtime-diagnostics-export.json>

Fixed replay entry (default sample input + fixed baseline mode):

pnpm run report:render-parity-runtime-replay

Fixed batch replay entry (directory -> per-input replay runs -> one dashboard):

pnpm run report:render-parity-runtime-replay-batch

Replay history/gate entry (dashboard -> rolling history + CI gate result):

pnpm run report:render-parity-runtime-replay-history-gate

Strict replay history/gate entry (fail process on gate failure):

pnpm run report:render-parity-runtime-replay-history-gate:strict

Stage-strict replay history/gate entry (strict + stage unknown thresholds):

pnpm run report:render-parity-runtime-replay-history-gate:strict:stage

Before executing the gate, this command clears its dedicated stage history/gate artifacts to prevent prior snapshots from contaminating rolling unknown and unknown-rate evaluation.

This command writes dedicated artifacts:

- `./docs/product-requirements/render-parity-reports/runtime-replay.stage.batch.history.json`
- `./docs/product-requirements/render-parity-reports/runtime-replay.stage.batch.gate.json`

Resource-rate-strict replay history/gate entry (strict + resource unknown-rate thresholds):

pnpm run report:render-parity-runtime-replay-history-gate:strict:resource-rate

Before executing the gate, this command clears its dedicated resource-rate history/gate artifacts to prevent prior snapshots from contaminating rolling unknown-rate evaluation.

This command writes dedicated artifacts:

- `./docs/product-requirements/render-parity-reports/runtime-replay.resource-rate.batch.history.json`
- `./docs/product-requirements/render-parity-reports/runtime-replay.resource-rate.batch.gate.json`

One-shot batch + gate entry:

pnpm run report:render-parity-runtime-replay-batch-gate

One-shot batch + strict gate entry:

pnpm run report:render-parity-runtime-replay-batch-gate:strict

One-shot batch + stage-strict gate entry:

pnpm run report:render-parity-runtime-replay-batch-gate:strict:stage

One-shot batch + resource-rate-strict gate entry:

pnpm run report:render-parity-runtime-replay-batch-gate:strict:resource-rate

## CI Baseline Gate Flow

PR baseline gate workflow:

- `.github/workflows/render-parity-strict-gate.yml`

Workflow behavior:

- always runs strict replay gate command:
  - `pnpm --filter @venus/vector-editor-web run report:render-parity-runtime-replay-batch-gate:strict`
- consumes gate artifact:
  - `apps/vector-editor-web/docs/product-requirements/render-parity-reports/runtime-replay.batch.gate.json`
- routes machine-readable failures by code (`RP_GATE_*`) as grouped CI annotations (`RenderParityGate`)
- fails PR job when strict gate step fails

## Default Output Directory

- ./docs/product-requirements/render-parity-reports

## Output Files

The command writes three files per run:

- <label>.<timestamp>.samples.json
- <label>.<timestamp>.report.json
- <label>.<timestamp>.summary.json

Where:

- label defaults to runtime-session
- timestamp is generated from report.generatedAt in a filename-safe format

## Optional Flags

- --output-dir <path>
- --label <name>
- --min-samples <number>
- --max-text-fallback <number>
- --max-deferred-image <number>
- --max-deferred-text <number>
- --baseline-summary <baseline-summary.json>
- --trend-dir <summary-dir>
- --replay-baseline

When `--baseline-summary` is provided, the auto report command also writes:

- <label>.<timestamp>.diff.json

and prints trend signals (`overallTrend`, dominant-reason change flags).

When `--trend-dir` is also provided, the auto report command additionally writes:

- <label>.<timestamp>.trend.json

and prints aggregate trend counters across baseline-vs-many summary comparisons.

When `--replay-baseline` is provided:

- baseline path is pinned to `<output-dir>/<label>.baseline.summary.json`
- first run bootstraps this baseline file from the current summary
- subsequent runs automatically produce diff against the pinned baseline
- trend comparison directory defaults to `--output-dir` when `--trend-dir` is not provided

This mode is intended for stable replay sessions that append one trend snapshot per run.

## Summary Diff Command

pnpm run report:render-parity-summary-diff -- --baseline <baseline-summary.json> --current <current-summary.json> --output <diff-report.json>

Use this command to compare two `.summary.json` files and detect:

- dominant reason changes (webgl/webgpu)
- rejected count deltas (known/unknown)
- overall trend signal (`improved`, `regressed`, `mixed`, `unchanged`)

## Summary Trend Command

pnpm run report:render-parity-summary-trend -- --baseline <baseline-summary.json> --dir <summary-dir> --output <trend-report.json>

Use this command to compare one baseline against all `.summary.json` files in a directory and aggregate:

- per-run diff rows (sorted by generatedAt)
- trend counters (`improved/regressed/mixed/unchanged`)

## Replay Batch Dashboard

`report:render-parity-runtime-replay-batch` scans one diagnostics input directory and executes replay mode once per JSON file.

Defaults in package script:

- input directory: `./scripts/runtime-diagnostics-replay-inputs`
- dashboard output: `./docs/product-requirements/render-parity-reports/runtime-replay.batch.dashboard.json`

Feature-level replay sample set (tracked in dedicated directory `./scripts/runtime-diagnostics-feature-replay-inputs`):

- `sample-feature-text.runtime-diagnostics.json` (known text capability gate pressure)
- `sample-feature-unknown.runtime-diagnostics.json` (unknown capability gate token coverage)

Dashboard includes:

- per-input run metadata (`label`, `summaryPath`, optional `diffPath/trendPath`)
- replay baseline bootstrap marker per row
- aggregate trend counters (`improved/regressed/mixed/unchanged/unknown`)
- aggregate feature-capability reject counters (`webgl/webgpu known + unknown`)
- aggregate frame-stage scheduler distribution (`interactive/normal/unknown`)
- aggregate frame-stage scene-apply distribution (`none/full-load/preview-load/incremental-patch/unknown`)
- aggregate runtime resource decode-status distribution (`queued/decoding/ready/failed/unknown`)
- aggregate runtime resource compression-codec distribution (`none/brotli/gzip/zstd/lz4/unknown`)

Contract coverage now includes:

- empty input directory rejection
- invalid diagnostics JSON rejection
- first-run replay baseline bootstrap -> `unknown` trend rows

## Replay History Gate

`report:render-parity-runtime-replay-history-gate` reads one batch dashboard and writes:

- `runtime-replay.batch.history.json`
- `runtime-replay.batch.gate.json`

History behavior:

- append/replace snapshot by `generatedAt`
- keep only latest `--window-size` snapshots (default `20`)
- aggregate rolling trend counters across retained snapshots
- aggregate rolling frame-stage scheduler counters across retained snapshots
- aggregate rolling frame-stage scene-apply counters across retained snapshots
- aggregate rolling runtime resource decode-status counters across retained snapshots
- aggregate rolling runtime resource compression-codec counters across retained snapshots

Gate behavior (latest snapshot based):

- fail when `regressed > --max-regressed` (default `0`)
- fail when `mixed > --max-mixed` (default `0`)
- fail when `unknown > --max-unknown` (default `0`)
- fail when `processedCount < --min-processed` (default `1`)

Rolling gate behavior (history window based, optional):

- fail when `rolling regressed > --max-rolling-regressed`
- fail when `rolling mixed > --max-rolling-mixed`
- fail when `rolling unknown > --max-rolling-unknown`

Feature-capability gate behavior (optional):

- fail when `latest webgl feature unknown > --max-webgl-feature-unknown`
- fail when `latest webgpu feature unknown > --max-webgpu-feature-unknown`
- fail when `rolling webgl feature unknown > --max-rolling-webgl-feature-unknown`
- fail when `rolling webgpu feature unknown > --max-rolling-webgpu-feature-unknown`

Runtime resource gate behavior (optional):

- fail when `latest runtime resource decode unknown > --max-resource-decode-unknown`
- fail when `latest runtime resource compression unknown > --max-resource-compression-unknown`
- fail when `rolling runtime resource decode unknown > --max-rolling-resource-decode-unknown`
- fail when `rolling runtime resource compression unknown > --max-rolling-resource-compression-unknown`
- fail when `latest runtime resource decode unknown rate (%) > --max-resource-decode-unknown-rate-percent`
- fail when `latest runtime resource compression unknown rate (%) > --max-resource-compression-unknown-rate-percent`
- fail when `rolling runtime resource decode unknown rate (%) > --max-rolling-resource-decode-unknown-rate-percent`
- fail when `rolling runtime resource compression unknown rate (%) > --max-rolling-resource-compression-unknown-rate-percent`

Frame-stage gate behavior (optional):

- fail when `latest stage scheduler unknown > --max-stage-scheduler-unknown`
- fail when `latest stage scene-apply unknown > --max-stage-scene-apply-unknown`
- fail when `rolling stage scheduler unknown > --max-rolling-stage-scheduler-unknown`
- fail when `rolling stage scene-apply unknown > --max-rolling-stage-scene-apply-unknown`
- fail when `latest stage scheduler unknown rate (%) > --max-stage-scheduler-unknown-rate-percent`
- fail when `latest stage scene-apply unknown rate (%) > --max-stage-scene-apply-unknown-rate-percent`
- fail when `rolling stage scheduler unknown rate (%) > --max-rolling-stage-scheduler-unknown-rate-percent`
- fail when `rolling stage scene-apply unknown rate (%) > --max-rolling-stage-scene-apply-unknown-rate-percent`

CLI fail-fast behavior:

- `--fail-on-status` returns a non-zero process exit code when gate status is `fail`
- when runtime resource unknown-rate dimensions exceed configured thresholds, CLI also prints:
  - `resource-rate-exceeded: <count>`
  - one `resource-rate: <failureCode>: actual=<value> threshold=<value>` row per exceeded dimension
- when stage unknown-rate dimensions exceed configured thresholds, CLI also prints:
  - `stage-rate-exceeded: <count>`
  - one `stage-rate: <failureCode>: actual=<value> threshold=<value>` row per exceeded dimension

Gate artifact includes:

- `status: pass|fail`
- `reasons`: human-readable failure reasons
- `failures`: machine-readable codes + messages for CI routing
- `stageUnknownRatePercentSummary`: structured latest/rolling stage unknown rate percentages for scheduler + scene-apply
- `stageUnknownRateViolationSummary`: structured latest/rolling stage unknown-rate threshold evaluation rows (`actualRatePercent`, `thresholdRatePercent`, `exceeded`, `exceededCount`)
- `rollingRuntimeResourceDecodeStatusCounter`: structured rolling decode-status totals (`queued/decoding/ready/failed/unknown`)
- `rollingRuntimeResourceCompressionCodecCounter`: structured rolling compression-codec totals (`none/brotli/gzip/zstd/lz4/unknown`)
- `resourceUnknownRatePercentSummary`: structured latest/rolling resource unknown rate percentages for decode + compression
- `resourceUnknownRateViolationSummary`: structured latest/rolling resource unknown-rate threshold evaluation rows (`actualRatePercent`, `thresholdRatePercent`, `exceeded`, `exceededCount`)

## Rate Fluctuation Replay Scenarios

Use replay-history-gate contracts for deterministic stage unknown-rate verification:

- latest scene-apply unknown-rate failure:
  - fixture profile: latest snapshot scene-apply `unknown=2`, total scene-apply stage rows `2`
  - threshold profile: `maxStageSceneApplyUnknownRatePercent=40`
  - expected: gate fail with `RP_GATE_LATEST_STAGE_SCENE_APPLY_UNKNOWN_RATE`
- latest scheduler unknown-rate failure:
  - fixture profile: latest snapshot scheduler `unknown=2`, total scheduler stage rows `2`
  - threshold profile: `maxStageSchedulerUnknownRatePercent=40`
  - expected: gate fail with `RP_GATE_LATEST_STAGE_SCHEDULER_UNKNOWN_RATE`
- rolling scheduler/scene-apply unknown-rate failure:
  - fixture profile: two snapshots in one history window, first known-only then unknown-only
  - threshold profile: `maxRollingStage*UnknownRatePercent=40`
  - expected: gate fail with `RP_GATE_ROLLING_STAGE_*_UNKNOWN_RATE` and
    `stageUnknownRatePercentSummary.rollingStage*UnknownRatePercent=50`
