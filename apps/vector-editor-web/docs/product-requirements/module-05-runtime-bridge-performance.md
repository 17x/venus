# 模块 05：Runtime 桥接、性能与稳定性

## 1. 模块目标

保证 Product 高频交互在复杂文档中仍稳定、低延迟、可诊断：

- Product <-> Runtime 命令桥
- 预览态与提交态一致性
- 高频路径性能优化
- 可观测性与故障恢复

## 2. 架构约束

1. Product 负责语义与策略，不直接实现底层运行能力。
2. Runtime 负责会话、命中、预览态、命令分发。
3. Engine 负责渲染与底层能力。

## 3. 需求条目

### R1 命令桥标准化

1. 所有产品动作统一映射到运行时命令。
2. 命令需要具备：

- 来源上下文
- 幂等语义
- 可回放信息

验收标准：

- 同一动作在不同入口（快捷键/菜单/手势）结果一致。

### R2 预览提交一致性

1. 拖拽类操作必须经过 preview state。
2. pointerup 后提交 patch，清理临时态。
3. commit 失败要有回退策略。

验收标准：

- 不出现“视觉已变但文档未变”或反向不一致。

### R3 性能预算

1. 高频交互（移动/缩放/旋转/路径拖拽）需定义性能预算。
2. 典型文件规模需给出明确容量档位（小/中/大）。
3. 命中与吸附策略支持动态预算降级。

验收标准：

- 典型场景交互无明显卡顿。
- 降级触发有日志可追踪。

### R4 可观测性

1. 增加运行时诊断面板（可选开发模式）。
2. 记录关键指标：

- 交互延迟
- 命中候选量
- 提交耗时
- 回滚次数

验收标准：

- 问题定位可在 10 分钟内完成首轮归因。

### R5 容错与恢复

1. 中断事件后必须保证状态可恢复。
2. 异常命令不得污染历史与文档。

验收标准：

- 可构造异常场景下仍可继续编辑。

## 4. 里程碑建议

1. 第一阶段：统一命令桥与诊断字段。
2. 第二阶段：性能预算与降级策略。
3. 第三阶段：可视化性能面板与自动回归基线。
